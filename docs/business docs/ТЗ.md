## 1. Общие сведения

### 1.1 Назначение системы

Публичный одностраничный сайт-витрина (лендинг) театрально-хорового коллектива, наполняемый через собственную административную панель (CMS). Сайт демонстрирует репертуар мюзиклов, ведёт к покупке билетов и принимает интерес к участию в хоре.

### 1.2 Бизнес-цели

1. **Продажа билетов** — привести зрителя к покупке (текущая реализация: переход по внешней ссылке, заложенной в карточке спектакля).
2. **Рекрутинг в хор** — привлечь новых участников коллектива (секция «Присоединиться» + кнопка «В хор» в мобильном баре).
3. **Премиальный имидж** — сформировать восприятие коллектива как профессионального, атмосферного, «театрального» через готическую визуальную подачу и моушн-эффекты.

### 1.3 Целевая аудитория

- **Зрители** — потенциальные покупатели билетов на мюзиклы (Москва и окрестности).
- **Кандидаты в хор** — люди, желающие петь и участвовать в постановках.
- **Контент-менеджер** — администратор, наполняющий сайт через CMS без участия разработчика.

### 1.4 Термины и глоссарий

| Термин | Значение |
|---|---|
| **Лендинг** | Публичная одностраничная витрина (актуально — `red_draft3_1_1`). |
| **CMS / Админка** | Панель управления контентом на `/admin`. |
| **Hero** | Первый («главный») экран лендинга с названием, фоном/видео и обратным отсчётом. |
| **Featured / текущий проект** | Спектакль, чьё название совпадает с заголовком Hero; если совпадения нет — первый в списке. |
| **Афиша** | Карусель карточек всех спектаклей. |
| **Солист** | Объект `{name, role, photo}` внутри спектакля. |
| **Медиатека** | Раздел CMS для управления всеми загруженными фото и видео. |
| **db.json** | Файл-хранилище всего контента сайта. |
| **CSRF-токен** | Токен защиты мутирующих запросов админки. |
| **reduced-motion** | Системная настройка пользователя «уменьшить движение» (`prefers-reduced-motion`). |

## 2. Архитектура и стек

### 2.1 Компоненты системы

| Компонент | Расположение | Описание |
|---|---|---|
| **Публичный фронт** | `/red_draft3_1_1/` (`index.html`, `script.js`, `style.css`) | Статический HTML + vanilla JS, рендерит контент из `/api/content`. |
| **Админ-CMS** | `/frontend/admin/` (`admin.html`, `js/admin.js`, `css/admin.css`) | Vanilla JS + Bootstrap 5.3, редактирует все секции и медиа. |
| **Бэкенд-API** | `/backend/server.js` | Express REST API: контент, авторизация, загрузка/оптимизация медиа. |
| **Хранилище контента** | `/backend/data/db.json` | JSON-файл со всем контентом (атомарная запись). |
| **Хранилище доступа** | `/backend/data/auth.json` | Логин + bcrypt-хэш пароля. |
| **Файловое хранилище** | `/backend/uploads/` (+ `thumbs/`, `videos/`) | Загруженные фото, превью, видео. |

> **Исторический контекст.** В репозитории присутствуют параллельные макеты-итерации (`v2`–`v5`, `red_draft`, `red_draft2`, `red_draft3`, `red_draft3_1`, `red_draft3_1_1`), все отдаются как статика с одного бэкенда и используют общий `/api/content`. Актуальной публичной версией считается **`red_draft3_1_1`**.

### 2.2 Технологии

| Слой | Технология |
|---|---|
| Бэкенд-сервер | Node.js + **Express** |
| Загрузка файлов | **Multer** (disk storage) |
| Обработка изображений | **Sharp** (даунскейл до 2560px, перекодирование, EXIF-поворот, превью 400×400) |
| Авторизация | **bcryptjs** (хэш пароля) + самописные сессии в памяти + CSRF-токены |
| Хранилище | JSON-файл (`db.json`) с атомарной записью (tmp + rename) |
| Публичный фронт | Чистый HTML + CSS + **vanilla JavaScript** (без сборки) |
| Шрифты | Druzhok (заголовки), Avenir Next (текст), Futura (лейблы); фолбэк — Playfair Display |
| Админ-фронт | **Bootstrap 5.3** + vanilla JavaScript |
| Карты | Яндекс.Карты (iframe map-widget) |

> **Примечание о расхождении со «стеком из CLAUDE.md».** Декларированный в `CLAUDE.md` стек (Next.js 14, TypeScript, Drizzle, NextAuth) **не используется**: реальная реализация — Express + JSON + vanilla JS без сборки. Это сознательное решение MVP-этапа (см. раздел 9 «Ограничения и риски»).

### 2.3 Схема взаимодействия (текстом)

```
[ Браузер зрителя ]
      │  GET /red_draft3_1_1/  (статика: html/css/js)
      │  GET /api/content      (весь контент одним JSON)
      │  GET /api/gallery/photos (список фото для галереи)
      ▼
[ Express (backend/server.js) ] ──── читает ──▶ [ db.json ]
      ▲                          ──── отдаёт ──▶ [ /uploads/* статикой ]
      │
[ Браузер админа ] ── /admin (Bootstrap SPA)
      │  POST /api/login         → token + csrfToken (в sessionStorage)
      │  PUT/POST/DELETE /api/*   (X-Auth-Token + X-CSRF-Token)
      │  POST /api/upload(/video) (Sharp оптимизация → uploads/)
      ▼
[ db.json + uploads/ ]  (атомарная запись JSON, проверка «файл используется»)
```

Поток данных: контент-менеджер правит данные в админке → бэкенд пишет в `db.json` → зритель при загрузке лендинга получает свежий `/api/content` → JS рендерит секции на клиенте.

### 2.4 Окружение и деплой

- **Запуск:** `npm start` из корня (бэкенд слушает `PORT`, по умолчанию `3000`).
- **Окружение:** переменные из корневого `.env` (`PORT` и т.п.).
- **Деплой:** Render free-tier. ⚠️ **Диск эфемерный** — загруженные через CMS медиа и правки `db.json` живут только до следующего деплоя (в git вернётся лишь закоммиченное). Подробнее — раздел 9.
- **Cold start:** на free-tier первый запрос после простоя ждёт ~30–50 с пробуждения инстанса.

## 9. Ограничения и риски

| Риск / ограничение | Описание | Митигация |
|---|---|---|
| **JSON вместо БД** | Контент в `db.json` — параллельные записи могут привести к гонкам; пользовательские заявки нельзя хранить в JSON. | Атомарная запись уже есть; заявки — только в БД (беклог M1). |
| **Render free-tier — эфемерный диск** | Загруженные медиа и правки CMS пропадают при деплое. | R2 для медиа + Postgres для данных (M1). |
| **Cold start** | Первый запрос после простоя ждёт 30–50 с. | Платный план или внешний «пингер». |
| **Vanilla без сборки** | Нет TypeScript/типобезопасности, нет бандлинга/трешейкинга, расхождение со стеком из `CLAUDE.md`. | Осознанный выбор MVP; миграция на Next.js — отдельная инициатива. |
| **Сессии в памяти** | Сбрасываются при рестарте/сне инстанса — админ разлогинивается. | Персистентные сессии (M6). |
| **152-ФЗ при формах** | Любая форма с email/телефоном требует политики конфиденциальности и согласия на обработку ПДн. | Cookie-баннер + политика до запуска формы заявки. |
| **Клиентский рендер и SEO** | Поисковики/мессенджеры не видят серверных мета. | Серверный рендер мета (M5). |
